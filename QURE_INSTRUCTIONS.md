# Qure + Vibe Audit Integration

This file tells the Qure AI agent how to work with Vibe Audit.

## What is Vibe Audit?

Vibe Audit is a multi-layer behavioral regression pipeline. It catches cases where tests pass but the user experience is broken. It uses your Qure-generated tests as primary evidence — both for analysis AND for actual execution.

## Quick Commands

| Command | What it does |
|---------|-------------|
| `node scripts/qure-run.js` | Full regression analysis |
| `node scripts/qure-run.js --suggest-tests` | Show what tests to create next |
| `node scripts/qure-run.js --check-coverage` | Show coverage map and gaps |

## How to create tests (best to worst)

### 1. Record a flow (BEST — highest quality tests)

Recording produces the most reliable tests because selectors come from the real DOM.

1. Click **Record** in the chat input
2. Navigate the app naturally (login, add to cart, checkout, etc.)
3. Click **Finish Recording**
4. Say: "Create a test from this recording. Add `// Generated by Qure` at the top. Save to `tests/qure/`"

**When to record:**
- Critical user journeys (login, checkout, payment)
- Flows that involve multiple pages
- Anything that handles money or user data

### 2. Describe a scenario (GOOD — fast for simple cases)

Just type what to test. Good for edge cases and validation checks.

```
Create a test that verifies: user cannot submit checkout form without filling name and email fields.
Add "// Generated by Qure" at the top. Save to tests/qure/. Run it to verify it passes.
```

### 3. Ask for coverage analysis (DISCOVERY — find what's missing)

```
Run node scripts/qure-run.js --suggest-tests
```

Then create tests for the suggestions.

## Where to put tests

- **Primary:** `tests/qure/` (organized by convention)
- **Also detected:** `tests/`, `e2e/`, `test/`, `__tests__/`, `spec/`
- Auto-detection looks for the `// Generated by Qure` marker comment

### Required marker

Always add this at the top of generated test files:

```typescript
// Generated by Qure — Flow: "Description of the user journey"
```

## Test conventions for this project

- Use Playwright (`@playwright/test`)
- Tests run against `http://localhost:3000` (Next.js demo app)
- Playwright config is at `demo/playwright.config.ts`
- Use `beforeEach` to clear cart state via `POST /api/cart/clear`
- Match the style in `tests/qure/checkout-edge-cases.spec.ts`

## What Vibe Audit does with your tests

| Layer | How your tests are used |
|-------|------------------------|
| Layer 1 (Static) | Selectors in your tests are cross-referenced against changed files |
| Layer 2 (Execution) | Your tests are RUN against the live app — pass/fail is evidence |
| Layer 3 (Visual) | Pages your tests visit are screenshotted for regression comparison |
| Layer 5 (Synthesis) | LLM correlates test results with code changes for risk scoring |

Tests in `tests/qure/` get **1.5x risk weight** — if a change breaks a recorded user flow, it's flagged as HIGH confidence.

## Flows to prioritize recording

These are the most valuable flows to protect (in order):

1. **Checkout end-to-end** — add item → cart → checkout → submit → confirmation
2. **Login → shop** — authenticate and return to catalog
3. **Cart operations** — add, remove, update quantity, verify totals
4. **Navigation** — can users reach all pages from the homepage?
5. **Form validation** — required fields, error messages, invalid input
6. **Empty states** — empty cart, no items, logged-out user accessing protected pages

## After creating tests

Run to verify the new tests improve coverage:

```
node scripts/qure-run.js --check-coverage
```
